# Video Action — техническая документация и план реализации

## 1) Суть проекта

Video Action — это платформа для автоматизации монтажа видео на основе AI.

Идея:
- получать от ассистента (Reverie) семантический объект сценария;
- строить из него структуру проекта (сцены, треки, нарратив, ассеты, сроки);
- формировать таймлайн и запускать рендер;
- возвращать готовый результат и управлять итерациями правок из одного приложения.

Важно: управление интерфейсом через CapCut напрямую как через официальный MCP API сейчас не берется за основу. Базовый путь — собственный backend-рендер с абстракцией через контракты и адаптеры.

## 2) Что уже есть в репозитории (Baseline)

- Монорепозиторий на `pnpm` + `turbo`.
- Две цели приложений:
  - `apps/web` — frontend (Next.js).
  - `apps/api` — backend (NestJS).
- Общие пакеты:
  - `packages/api` — общий API/контрактный слой.
  - `packages/typescript-config` — единые ts-конфиги.
- Единый стандарт качества:
  - линтер/форматтер через `Biome`;
  - hooks через `lefthook` + `commitlint`;
  - CI workflows (`checks.yml`, `deploy.yml`).
- Базовые Dockerfile для `web` и `api`.

## 3) Задачи продукта

1. Дать PM/редакторам инструмент визуальной сборки сценария в структуру видео.
2. Сохранять семантический объект как единый источник истины (single source of truth).
3. Автоматизировать сборку таймлайна и рендер без ручной ручной склейки.
4. Поддерживать версии и историю правок проекта (чтобы можно было откатываться и восстанавливать пайплайн).
5. Подготовить инфраструктуру для плагинов: новые источники контента, новые форматы сцены, новые способы экспорта.

## 4) Архитектурные принципы

1. **Backend-first рендеринг**
   - UI только формирует и отправляет заявку.
   - Сложная обработка, сборка и рендер выполняются на сервере.

2. **Единый контракт проекта**
   - Все подсистемы работают с одним `VideoProject`-контрактом.
   - Версионирование контракта (`v1`, `v2`) обязателен.

3. **Разделение слоев**
   - `UI` — управление, предпросмотр, редактирование структуры.
   - `Application API` — валидация, авторизация, управление очередями.
   - `Execution Layer` — запуск рендер-джобов и контроль статуса.
   - `Adapters` — отдельный уровень под внешние инструменты рендеринга.

4. **Асинхронный процесс**
   - Рендеринг как фоновая задача (job, retry, retries, dead-letter/ошибки).

5. **Результат как артефакт**
   - Для каждого проекта и версии храним:
     - входные ассеты и параметры;
     - промежуточный state таймлайна;
     - сгенерированные файлы;
     - лог выполнения.

6. **Нормализованный плагинный подход**
   - Плагины подключаются через четкие интерфейсы (scene provider, asset provider, render target).

### Инфраструктурный/репозиторный

### Frontend

- `NestJS`
- TypeScript

### Деплой/операции
- Docker multi-stage

## 6) Карта пакетов (назначение)

### `apps/web`
Роль:
- интерфейс планирования и редактирования сценариев;
- форма/таймлайн/просмотр черновых артефактов;
- запуск и контроль рендер-запросов.

### `apps/api`
Роль:
- REST/GraphQL API (по выбору на раннем этапе);
- orchestration сценариев и заданий;
- валидация контрактов;
- запуск задач рендера и чтение их статусов.

### `packages/api`
Роль:
- общий контрактный слой и DTO между `web` и `api`;
- типы для плагинов, и единый формат обмена с Reverie.

### `packages/typescript-config`
Роль:
- единая типовая конфигурация TypeScript для всех частей.

## 7) План реализации и этапы (с критериями готовности)

### Этап 0. Базовый каркас (готово)
Цель:
- иметь однотипную инфраструктурную основу под стандарты компании.
Критерии готовности:
- репозиторий собирается через `pnpm`/`turbo`;
- lint и typecheck проходят по базовым пакетам;
- есть рабочие конфиги CI.

### Этап 1. Предпроектная модель и контракт
Цель:
- зафиксировать data model до начала реализации интерфейса.
Что делает команда:
- описать `VideoProject` (версии, сцены, дорожки, ассеты, эффекты, экспорт);
- описать `RenderRequest`, `RenderResult`, `RenderJob`.
Критерии готовности:
- все интерфейсы зафиксированы в `packages/api`;
- добавлены валидации и тестовые примеры контрактов;
- ревью согласования с бизнесом выполнено.

### Этап 2. API v1 (оркестрация заданий)
Цель:
- получать проект из UI, ставить рендер в очередь, отдавать статус.
Что делает команда:
- создать endpoints:
  - `POST /projects`
  - `POST /render-jobs`
  - `GET /render-jobs/:id`
  - `GET /exports/:id`
- внедрить статусную модель задания (`queued`/`running`/`done`/`failed`).
Критерии готовности:
- API принимает один тип входного контракта;
- рендер джобы создаются и переходят по статусам;
- ошибки пишутся в структуру статуса.

### Этап 3. Модуль конвертации из семантики в timeline
Цель:
- из семантического объекта собирать промежуточную сценическую модель.
Что делает команда:
- адаптер `ReverieAdapter` (черновой, локальный/через контракт);
- builder, который строит `TimelineDraft`.
Критерии готовности:
- есть минимум 3 типовых кейса (например: hook, 1 мин. скрипт, пост в соцсети);
- для каждого кейса видно ожидаемый timeline;
- корректность отображается в API-ответах.

### Этап 4. Frontend редактор v1 (таймлайн и пресеты)
Цель:
- дать оператору править структуру до рендера.
Что делает команда:
- список сцен;
- базовый предпросмотр/сводки;
- редактирование длительности/текста/переходов;
- кнопка `Запустить сборку`.
Критерии готовности:
- пользователь может создать проект, поправить структуру и запустить рендер;
- проект корректно отображается в UI и в API.

### Этап 5. Рендер-адаптер
Цель:
- реализовать первичный backend исполнитель сценария.
Что делает команда:
- запустить конвертацию в временный формат для рендера;
- выполнить локальный/контейнерный render pipeline (стартовый слой).
Критерии готовности:
- рендер создает mp4 для базового кейса;
- есть артефакты и ссылка на результат;
- хранение файлов детерминировано по `projectId`/`versionId`.

### Этап 6. Продуктовый минимум (v1.0)
Цель:
- рабочая версия для пилота.
Что делает команда:
- финальная структура проекта, экспорт пресетов, история версий;
- права доступа (минимум редактор/читатель);
- логирование и дашборд ошибок.
Критерии готовности:
- сценарий "идея → проект → рендер → финальный файл" проходит end-to-end;
- 3 сценария пилотного клиента проходят стабильно;
- критические ошибки обработаны и имеют понятный статус.

### Этап 7. Плагины и расширение экосистемы
Цель:
- закрыть расширяемость и кастомизацию.
Что делает команда:
- определить plugin API;
- подключить минимум два источника контента;
- один экспортный адаптер (например, разные профили качества).
Критерии готовности:
- новый плагин запускается без изменений core;
- есть документация по созданию плагина;
- регрессионные проверки не падают на существующих сценариях.

## 8) Рекомендуемые первые задачи для следующего спринта

1. Сформировать и согласовать контракт `VideoProject` (v1.0).
2. Подготовить API сущности `Project` и `RenderJob`.
3. Реализовать endpoint `POST /projects` + `GET /projects/:id`.
4. Добавить smoke-потоки в `apps/api` под стартовую очередь рендера.
5. Добавить минимальный UI `create project` и `run render`.

## 9) Принятые решения (кратко)

- Монорепозиторная модель с общими конфигами в стиле SecuritySphere.
- Web + Nest как ядро MVP.
- Рендер вне браузера (backend/серверные воркеры).
- Начинать с 1-2 надежных внутренних рендер-пайплайнов, не зависеть от непубличных API внешних редакторов.
- Планировать плагинность после стабилизации v1.
